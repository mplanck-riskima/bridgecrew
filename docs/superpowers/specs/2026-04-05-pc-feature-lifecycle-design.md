# PC Feature Lifecycle Design

**Date:** 2026-04-05
**Status:** Approved

## Problem

When working in Claude CLI directly on the PC running the Discord bot, there is no equivalent of the bot's feature tracking workflow. Features started in Discord aren't visible in the terminal, and vice versa. Setting up a new PC requires manually recreating any conventions. If the feature lifecycle rules change in the bot, the terminal workflow has no way to stay in sync.

## Goal

When using Claude CLI on any PC, it has the same feature lifecycle discipline as Discord bot sessions — tracking features, switching features, completing with summaries — with a one-command setup that stays in sync as the bot evolves.

## Design

### Source of Truth

The bot repo (`bridgecrew/`) is the single source of truth for feature lifecycle rules. A new canonical document, `docs/feature-lifecycle.md`, describes the lifecycle in full: status values, workflow steps, summary format, and conventions. This doc drives the CLI command generator so that any change to the lifecycle is automatically reflected in the PC commands on next setup.

### Architecture

```
bridgecrew/                          (source of truth)
  docs/
    feature-lifecycle.md             ← canonical lifecycle rules
  scripts/
    generate_claude_commands.py      ← renders command .md files from lifecycle doc
  setup-claude-pc.sh                 ← run on any new PC to install/update

<project>/                           (per-project, shared by bot AND CLI)
  .claude/
    features/
      feature-index.json             ← index, current_feature, session map
      <feature-name>.json            ← per-feature state record
      _archived/                     ← archived state for completed/discarded features
  features/
    <feature-name>.md                ← narrative feature docs (unchanged)
  CLAUDE.md                          ← project instructions (unchanged)

~/.claude/                           (user-level, generated by setup-claude-pc.sh)
  CLAUDE.md                          ← global feature lifecycle rules (merged via marked block)
  commands/
    start-feature.md
    resume-feature.md
    complete-feature.md
    list-features.md
    discard-feature.md
```

### State Format

Feature state moves from `<project>/.claude-bot/features/` to `<project>/.claude/features/`. The JSON schema is unchanged — this is a path-only refactor.

**`feature-index.json`** — lightweight index for the project:
```json
{
  "current_feature": "my-feature",
  "sessions": {
    "<session-uuid>": {
      "feature": "my-feature",
      "session_start": "2026-04-05T10:00:00Z"
    }
  }
}
```

**`<feature-name>.json`** — full state record for one feature:
```json
{
  "name": "my-feature",
  "status": "active",
  "session_id": "<uuid>",
  "subdir": null,
  "started_at": "2026-04-05T10:00:00Z",
  "completed_at": null,
  "sessions": [],
  "total_cost_usd": 0.0,
  "total_input_tokens": 0,
  "total_output_tokens": 0,
  "bridgecrew_feature_id": null
}
```

Status values: `active`, `completed`, `discarded`.

Because both the bot and CLI read/write the same `<project>/.claude/features/` path, feature state is naturally unified. A feature started in Discord appears in the terminal, and vice versa. No sync mechanism is needed.

### CLI Commands

Five slash commands are installed to `~/.claude/commands/`. Each is a markdown file containing instructions that Claude follows. Commands detect the current project from the working directory (`$PWD`).

| Command | Behavior |
|---|---|
| `/start-feature <name>` | Auto-completes any active feature (writes summary), then starts the new one |
| `/resume-feature` | Lists all features with status; user picks one; auto-completes any currently active |
| `/complete-feature` | Marks active feature done, generates git-diff-based summary to `features/<name>.md` |
| `/list-features` | Prints all features with status, subdir, started_at; marks active |
| `/discard-feature` | User picks feature; removes tracking, archives doc, strips from CLAUDE.md |

**Auto-complete behavior** (mirrors the bot): when a new feature is started or resumed while another is active, the displaced feature is automatically completed and a summary is generated before proceeding. There is no "paused" state.

**Summary generation** (on `/complete-feature` or auto-complete): Claude reads the feature's git diff since `started_at`, any existing `features/<name>.md`, then writes a structured summary back to `features/<name>.md`. This matches the bot's `run_feature_summary_prompt` behavior.

### Generator Script

`scripts/generate_claude_commands.py` is a pure Python stdlib script (no dependencies). It:

1. Reads `docs/feature-lifecycle.md` to extract lifecycle rules
2. Renders each of the five command `.md` files, injecting the relevant rule sections into each command's instruction text
3. Writes rendered files to a target directory (default: `~/.claude/commands/`)

Running the generator is idempotent — it always produces the same output for the same input.

### Setup Script

`setup-claude-pc.sh` is the single entry point for any new PC:

```bash
./setup-claude-pc.sh
```

Steps:
1. Checks Claude CLI is installed (warns if not found)
2. Runs `scripts/generate_claude_commands.py`
3. Copies rendered commands to `~/.claude/commands/` (creates dir if needed, overwrites)
4. Writes/merges feature lifecycle rules into `~/.claude/CLAUDE.md` via a marked block:
   ```
   # BEGIN: feature-lifecycle (auto-generated)
   ...
   # END: feature-lifecycle
   ```
   If the file exists, only the marked block is replaced. User customizations outside the block are preserved.
5. Prints a summary of what was installed/updated

**New PC flow:**
```bash
git clone <repo>
cd bridgecrew
./setup-claude-pc.sh     # installs Claude commands + global CLAUDE.md
./setup.sh               # installs bot (existing, unchanged)
```

**After lifecycle changes:**
```bash
./setup-claude-pc.sh     # regenerates and reinstalls everything
```

### Bot Migration

No bot code changes are needed. `core/state.py` already writes feature files to `<project>/.claude/features/`. The `.claude-bot/` directory retains only bot-runtime files that are not relevant to the CLI:

- `system_prompt.md` — static instruction cache
- `persona.md` — active persona
- `append_system_prompt.md` — combined prompt file
- `sessions/` — per-thread session prompt files

Feature state has already been at `<project>/.claude/features/` — the CLI commands can read and write it today.

## New Files

| File | Purpose |
|---|---|
| `docs/feature-lifecycle.md` | Canonical lifecycle rules — the source of truth |
| `scripts/generate_claude_commands.py` | Renders CLI command files from lifecycle doc |
| `setup-claude-pc.sh` | One-command setup for any PC |

## Changed Files

None — no existing bot code needs to change.

## Out of Scope

- Migration of existing `.claude-bot/features/` data to `.claude/features/` (existing projects can re-create state by starting/resuming features; historical data is not critical)
- Dashboard integration changes (bridgecrew_feature_id fields are unchanged)
- Any changes to Discord bot command behavior or UI
